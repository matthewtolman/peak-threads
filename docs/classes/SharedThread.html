    Class SharedThread

    Represents a shared thread. Shared threads are shared across browser contexts (tabs, windows, iframes). They are unique per url, so you can't spin up multiple of the same shared thread. Additionally, since they're shared they cannot be controlled as tightly as a normal Thread.

    They're useful for resource consolidation, as well as for sharing data across contexts (e.g. caches, thread pools). However, be careful as you need to use cache busting to "upgrade" them! In other-words, you need a new URL whenever the code changes, so you'll have a shared thread per version not per client.

    Shared threads have a lot of oddities due to the oddities of the underlying SharedWorker API. For instance, we "connect" to them with ports and we "disconnect" by closing the port not the thread. The help reinforce this, the API is intentionally different from the Thread API since concepts do not map across.

    Also, there is no "thread pool" for shared threads since we can't pool them. Instead, you'd create a single "pool" shared thread and have the thread pool be inside the shared thread. Then you'd talk to the shared thread which then delegates to the pool.

    Also, handlers in a shared thread always get an extra parameter - the Connection representing who they're talking to. You will need to call Connection.postMessage, Connection.transfer, and so on rather than a global postMessage or a global transfer.

    Finally, you'll need to use registerSharedHandler rather than registerHandler for registering shared thread handlers. This does mean that the "thread" and "shared thread" code is not compatible across each other. That's part of the Web API design, and instead of trying to hide it (most likely by adding a "connection" to the Thread-side) we decided to embrace it. Again, the semantics are different, how things work are different, and if you're not aware then you will run into issues.
